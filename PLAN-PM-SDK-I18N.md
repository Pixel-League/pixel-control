# PLAN - Pixel MatchMaking SDK, i18n & Error Pages (2026-03-07)

## Context

- **Purpose**: Set up the API client SDK generation, internationalization, and error pages for the Pixel MatchMaking platform. This is PLAN 3 of Phase 1 from `ROADMAP-PIXEL-PLATFORM.md`.
- **Scope**: 4 features from domain D0 (Foundation): D0.3 (SDK generation), D0.4 (API client wrapper), D0.8 (i18n with next-intl), D0.9 (error pages & fallbacks).
- **Non-goals**: Actual matchmaking logic, real API calls to pixel-control-server endpoints (no server required to be running), WebSocket setup (D10.1), profile sync (D1.6), roles enforcement (D1.8). These come in subsequent PLANs.
- **Background / Findings**:
  - Swagger JSON endpoint: `http://localhost:3000/api/docs-json` (confirmed in `pixel-control-server/src/main.ts:39-41`). Note: global prefix is `v1`, but Swagger is mounted at `/api/docs` (outside prefix).
  - Two SDK generation options evaluated:
    - **`openapi-typescript-codegen`**: Generates a vanilla TypeScript client with typed request/response interfaces. Lighter, no runtime dependencies. Better for our use case — we wrap with our own API client layer (D0.4).
    - **`orval`**: Generates TanStack Query hooks. Heavier, opinionated. Overkill for Phase 1 where we need a simple SDK.
  - Decision: Use **`openapi-typescript-codegen`** for simplicity. Can switch to orval later if TanStack Query hooks are desired.
  - **next-intl** is the i18n library specified in the roadmap. Requires: `messages/fr.json`, `messages/en.json`, `next-intl` plugin in `next.config.ts`, `i18n/request.ts` config, middleware for locale detection.
  - 64+ hardcoded French strings across 10 files need extraction: 5 placeholder pages, `AppTopNav.tsx`, `SignInForm.tsx`, `navigation.ts`, `layout.tsx` metadata, `auth/signin/page.tsx`.
  - Current pages have missing French accents (e.g., `competitif` should be `compétitif`, `creez` should be `créez`). i18n translation files will fix these.
  - DS provides `Skeleton` component for loading states. `Card`, `Button`, `Alert` for error pages.
  - Next.js 16.1: error pages use `error.tsx` (client component), `not-found.tsx` (server component), `loading.tsx` (server component).
- **Constraints / assumptions**:
  - SDK generation requires pixel-control-server to be running locally. The `sdk:generate` script must fail gracefully when the server is unavailable.
  - Generated SDK code should be committed (not gitignored) so the app can build without the server running.
  - next-intl with App Router: uses `createNavigation()` for locale-aware `<Link>`, `useRouter`, `usePathname`. Default locale (FR) has no URL prefix.
  - All existing unit tests (73 from PLANs 1-2) must continue passing.
  - All auth pages and components must be updated to use translation keys.
  - DS components for error/loading states: `Skeleton`, `Card`, `Button`, `Alert`.
  - Webpack bundler (not Turbopack) — `--webpack` flag in dev/build scripts.
- **Environment snapshot**: Branch `feat/pm-auth-sso`, commit `4a56337`. New branch: `feat/pm-sdk-i18n`.
- **Dependencies**: `openapi-typescript-codegen` (dev), `next-intl`.

### Phase 2 - API client wrapper (D0.4)

- [Done] P2.1 - Create API client module
  - Create `src/lib/api/client.ts`:
    - Export a configured API client instance using the generated SDK.
    - Base URL from `config.apiUrl` (env `NEXT_PUBLIC_API_URL`).
    - Default headers: `Content-Type: application/json`, `Accept: application/json`.
    - Auth header injection placeholder (for future use — reads token from session).
    - Centralized error handling: wrap API errors in a typed `ApiError` class.
  - Create `src/lib/api/errors.ts`:
    - `ApiError` class with `status`, `message`, `data` fields.
    - `isApiError()` type guard.
- [Done] P2.2 - Add retry logic
  - Create `src/lib/api/retry.ts`:
    - `fetchWithRetry()` wrapper that retries on network errors and 5xx responses.
    - Max 3 retries with exponential backoff (200ms, 400ms, 800ms).
    - Only retry idempotent methods (GET, PUT, DELETE) — never retry POST.
  - Integrate retry logic into the API client.
- [Done] P2.3 - Create API client barrel export
  - Create `src/lib/api/index.ts` that re-exports the client, errors, and generated types.
  - This is the single import point for API usage across the app: `import { apiClient, ApiError } from '@/lib/api'`.
